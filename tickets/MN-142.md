The self-host Caddy only listened on `:80`, forcing Cloudflare into "Flexible" SSL
mode (unencrypted Cloudflare↔origin, and a footgun: switching the zone to Full/Full-strict
yields a 521 because nothing answers on origin :443).

Add an **optional, secret-free** origin-TLS drop-in so operators can run Full (strict):

- `docker/Caddyfile` now ends with `import /etc/caddy/tls.d/*.caddy` (matches nothing by
  default → still plain HTTP :80 for the OSS default), and sets `auto_https disable_redirects`
  so `:80` and `:443` stay independent (no Cloudflare redirect loop).
- `docker/tls.d/origin.caddy.example` — copy to `origin.caddy` on the server to serve `:443`
  with a Cloudflare Origin CA cert.
- `docker-compose.yml` mounts `./docker/tls.d` and `./docker/certs` into the caddy container.
- `.gitignore` excludes `docker/certs/*` and `docker/tls.d/*.caddy` — real certs/keys and the
  live snippet are never committed.

Validated with `caddy validate` in both modes (default :80-only, and TLS active).

## Acceptance
- [x] Default config (no snippet) still serves plain HTTP on :80.
- [x] With snippet + Origin cert, Caddy serves TLS on :443; no HTTP→HTTPS auto-redirect.
- [x] Certs/keys and the live snippet are gitignored.
- [ ] Production (storyos.dev): Origin cert installed, snippet active, zone set to Full (strict), verified.

Follow-up to the 521 incident (zone was on Full while origin only served :80).
