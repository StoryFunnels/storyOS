The view config read-mapper (`cleanConfig` in `apps/api/src/views/views.service.ts`) only
whitelisted a subset of config keys. It dropped `start_date_field_id`, `end_date_field_id`,
`card_size`, and `form` on every read — so a Timeline view could never load its date fields
after a page refresh (it showed "This timeline has no start-date field") and Form/card-size
config silently vanished. Values were stored correctly on write but lost on read.

Fix: pass `start_date_field_id`, `end_date_field_id`, `card_size`, and `form` through the
mapper (date fields still validated against live fields).

## Acceptance
- [x] A Timeline view persists its start/end date fields across reloads and renders bars.
- [x] Form and card_size config survive a read round-trip.
- [x] Deleted date fields are still dropped from timeline config on read.

Found while seeding the Northwind demo workspace for the marketing site screenshots.
