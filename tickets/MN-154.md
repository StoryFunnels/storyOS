## Why
Polished connector UIs (claude.ai, ChatGPT, the Customize tab) drive remote MCP servers over
**OAuth 2.1**, not a hand-pasted `Authorization: Bearer` header. Today mcp.storyos.dev is
PAT-only, so those UIs can't connect without editing config. Add OAuth so cloud users click
"Connect" and log in — while **PATs stay** for self-hosted / power / n8n use.

## Why it's very doable (foundation already here)
- Auth is **better-auth 1.6.23**, which ships the **`oidc-provider`** plugin (+ `jwt`, `bearer`).
  So the StoryOS API can act as the OAuth **Authorization Server** — `/authorize`, `/token`,
  discovery metadata, PKCE, consent, and **Dynamic Client Registration** — with users signing
  in via their existing StoryOS session.
- `packages/mcp/src/http.ts` already extracts a `Bearer` token; it just needs a second
  validation branch.

## Architecture (MCP authorization spec, 2025-06-18)
1. **Authorization Server** = better-auth `oidc-provider` on app.storyos.dev:
   `/.well-known/openid-configuration` + `/.well-known/oauth-authorization-server` (RFC 8414),
   `/authorize` (StoryOS login + a consent screen), `/token`, and **DCR `/register`** (RFC 7591)
   so connectors self-register. PKCE required.
2. **Protected-resource metadata** on mcp.storyos.dev:
   `/.well-known/oauth-protected-resource` (RFC 9728) → points clients at the AS. Add a Caddy/route
   for it (the mcp service serves it).
3. **Dual auth in `http.ts`** — on each request's `Bearer <token>`:
   - `mn_pat_…` → existing PAT path (self-managed) — **unchanged**.
   - otherwise → validate as an **OAuth access token** (verify JWT via the AS's JWKS from the `jwt`
     plugin, check audience = the MCP resource per RFC 8707, check scope), resolve the user, build
     the per-request client.
4. **Scopes / consent**: at minimum `storyos.mcp`; consent screen shows the app + which
   workspace(s). (Workspace-scoped tokens are a nice follow-up.)

## Keep PAT (self-managed)
Self-hosters run without any OAuth infra: `STORYOS_TOKEN=mn_pat_…` on the MCP, or the header in a
config. The dual-auth branch means both work against the same endpoint forever.

## Security must-haves
PKCE mandatory · exact redirect-URI matching · short-lived access tokens + refresh · audience-bound
tokens (a token minted for the app API must not be accepted by the MCP unless aud matches) · HTTPS
only · rotate signing keys · rate-limit `/token` and `/register`.

## MVP → ship
1. Enable `oidc-provider` + `jwt` plugins; expose discovery + `/register`; minimal consent page.
2. Serve `/.well-known/oauth-protected-resource` on mcp.storyos.dev.
3. Add the OAuth-token branch to `http.ts` (JWKS verify + aud/scope check).
4. **Test end-to-end from a real client** (claude.ai custom connector → login → consent → tools work).
5. Docs: "Connect via OAuth (cloud)" vs "Connect with a PAT (self-managed)".

## Built so far (flag-gated — prod unaffected until enabled)
better-auth ships a purpose-built **`mcp` plugin** that IS the AS (discovery, `/authorize`,
`/token`, DCR, token validation), so this is a wiring job, not a from-scratch OAuth server.
- API `auth.ts`: `mcp({ loginPage, oidcConfig:{ requirePKCE, allowDynamicClientRegistration, scopes }})`
  added, gated behind **`MCP_OAUTH`** env (default off). PAT auth unchanged.
- MCP `http.ts`: serves `/.well-known/oauth-protected-resource` (RFC 9728) → the AS, and returns
  `WWW-Authenticate: … resource_metadata=…` on 401 — both only when `MCP_OAUTH=true`. Any Bearer is
  forwarded to the API (PAT or OAuth token), so the MCP is auth-method-agnostic.
- New env `MCP_OAUTH` (api) + `MCP_OAUTH`/`MCP_PUBLIC_URL`/`MCP_AUTH_SERVER` (mcp).
- Typecheck + lint + build all green.

## To enable (remaining — needs the server + a real client)
1. **Migrate the OAuth tables**: `oauthApplication`, `oauthAccessToken`, `oauthConsent` (from the
   oidc-provider plugin) — add to `auth-schema.ts` and a drizzle migration (generate via the
   better-auth CLI with the plugin on, or hand-write to match the plugin's schema).
2. **Consent page**: the plugin renders a default consent screen; optionally build a branded
   `/oauth/consent` on the web and set `oidcConfig.consentPage`.
3. **Deploy + flip on**: set `MCP_OAUTH=true` on the api and mcp services; run migrations.
4. **End-to-end test** (the decisive step): add a claude.ai custom connector to
   `https://mcp.storyos.dev/mcp` → it discovers the AS → login → consent → tools work. Confirm a
   `mn_pat_` still works too, and that an expired/bogus OAuth token is rejected.

## Note
Security-critical. The scaffolding is in and inert; do NOT flip `MCP_OAUTH=true` in prod until the
migration is applied and the full connector flow is tested end-to-end.
