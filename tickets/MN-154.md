## Why
Polished connector UIs (claude.ai, ChatGPT, the Customize tab) drive remote MCP servers over
**OAuth 2.1**, not a hand-pasted `Authorization: Bearer` header. Today mcp.storyos.dev is
PAT-only, so those UIs can't connect without editing config. Add OAuth so cloud users click
"Connect" and log in â€” while **PATs stay** for self-hosted / power / n8n use.

## Why it's very doable (foundation already here)
- Auth is **better-auth 1.6.23**, which ships the **`oidc-provider`** plugin (+ `jwt`, `bearer`).

  discovery metadata, PKCE, consent, and **Dynamic Client Registration** â€” with users signing

- `packages/mcp/src/http.ts` already extracts a `Bearer` token; it just needs a second
  validation branch.

## Architecture (MCP authorization spec, 2025-06-18)
1. **Authorization Server** = better-auth `oidc-provider` on app.storyos.dev:
   `/.well-known/openid-configuration` + `/.well-known/oauth-authorization-server` (RFC 8414),
   `/authorize` (StoryOS login + a consent screen), `/token`, and **DCR `/register`** (RFC 7591)
   so connectors self-register. PKCE required.
2. **Protected-resource metadata** on mcp.storyos.dev:
   `/.well-known/oauth-protected-resource` (RFC 9728) â†’ points clients at the AS. Add a Caddy/route
   for it (the mcp service serves it).
3. **Dual auth in `http.ts`** â€” on each request's `Bearer <token>`:
   - `mn_pat_â€¦` â†’ existing PAT path (self-managed) â€” **unchanged**.
   - otherwise â†’ validate as an **OAuth access token** (verify JWT via the AS's JWKS from the `jwt`
     plugin, check audience = the MCP resource per RFC 8707, check scope), resolve the user, build
     the per-request client.
4. **Scopes / consent**: at minimum `storyos.mcp`; consent screen shows the app + which
   workspace(s). (Workspace-scoped tokens are a nice follow-up.)
