## Symptoms
Inviting a **member** showed `Invite failed — guests need at least one space`, yet the
person was invited (row created, email sent). Retrying stacked duplicate pending invites.

## Causes
1. **Misleading toast**: the frontend `onError` hardcoded the guest-space message for
   *every* invite error, regardless of the real cause.
2. **Non-atomic create**: `InvitesService.create` inserted the invite row, then `await`ed
   `sendMail`. Before the sender domain was verified in Resend, `sendMail` threw → the
   request 500'd *after* the row was committed → false error + a persisted invite. Each
   retry added another row.

## Fixes
- `mail/mailer.ts` — `sendMail` is now best-effort: a send failure is logged, not thrown,
  so it never breaks the calling flow (invites, verification, resets). The row is the
  source of truth and the accept link is returned regardless.
- `invites.service.ts` — re-inviting the same address now **refreshes the existing pending
  invite** (new token) instead of inserting a duplicate.
- members page — `onError` surfaces the real server error message.

## Acceptance
- [x] A member invite succeeds without a false error even if email delivery fails.
- [x] Re-inviting the same email updates the pending invite instead of duplicating it.
- [x] Error toast shows the actual reason.
- [ ] Deployed (rebuild api + web); revoke the stray duplicate invites in the UI.
