## Problem
Records carry `created_at` / `updated_at` / `created_by`, but the timestamps aren't
usable/visible like `created_by` is — you can't reliably see a creation date, add it as a
column, sort by it, or drive a calendar/timeline from it. "We don't have any dates."

## What
Make **created_at** and **updated_at** first-class **date-time** system fields on **every
database, system-wide** — parallel to the existing `created_by` system field:
- Auto-present on every database (existing + newly created + template + imported).
- Addable to any view (column / card chip), **sortable**, and usable as the date field of
  Calendar / Timeline views.
- Rendered as date or date-time (respect a display option), read-only (system-managed).
- Backfill: ensure existing databases (incl. Linear-imported) expose them and that values
  are populated (imports should set created_at to the source's created time where available).

## Acceptance
- [x] created_at + updated_at are selectable, sortable, toggleable date-time fields on every database.
- [x] Shown in table/board (toggleable), rendered as formatted dates, usable as calendar/timeline date axis; read-only.
- [x] Values sourced from the record timestamp (never blank). NOTE: for Linear-imported rows this is the import time, not the original Linear time — backfilling source timestamps is a follow-up.

Note: this is a cross-cutting change (schema/system-field defaults + view field pickers +
import mapping), hence M not S.
