## Why
The MCP could read schema and CRUD records but not **build** — no way for an agent to
create a database, add/modify fields, or manage views. So agents could fill a workspace,
not design one. The API already exposes all of this; the MCP just lacked the tools.

## What
Seven new tools in `packages/mcp/src/tools.ts` (20 total), all with name→id resolution and
validation-as-teacher (a bad field/view name returns the available options):

- `create_database` — new database in a space (resolves space by name).
- `add_field` — any creatable type; select/multi_select take option **labels**.
- `update_field` — rename and/or add select options.
- `delete_field` — soft-delete.
- `create_view` — table/board/calendar/gallery/list/feed/timeline/form; friendly params
  (`group_by`, `card_fields`, `date_field`, `start/end_date_field`) resolved to field ids
  and packed into the view config.
- `update_view` / `delete_view`.

Smoke-tested end-to-end against a live workspace: create db → add select field → create a
board grouped by it → describe confirms → bad field name rejected with guidance. README +
smoke tool-count updated.

## Acceptance
- [x] Agent can create a database, add fields (incl. select options), and build views.
- [x] Field/view references accept human names; errors list valid options.
- [x] stdio + hosted HTTP both expose the new tools (same registerTools).
- [ ] Deployed: rebuild the mcp container on the server.

## Fast-follow (separate)
- `create_relation` — link two databases (paired relation fields) so agents can model
  connected data, not just flat tables. Relations use a different endpoint than fields.
- `create_space` and `change_field_type`.
