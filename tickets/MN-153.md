## Why this is urgent (day-1 API contract)
Agents (MCP), no-code tools (n8n), and humans reference databases by **name / api_slug** —
UUIDs are opaque, so nobody types them. Today two databases named "Projects" in different
spaces resolve to `projects` and `projects_2`. That is a **silent wrong-target hazard**:
a flow that says "create a task in Projects" can hit the stale/other one. Once external
automations bind to these slugs, changing them is a breaking change — so we must get the
scheme right **before** anything depends on it.

Observed: JCM Agency has `Projects` (`projects`) and `Projects` (`projects_2`) — one likely a
stale duplicate from the Linear import, but the collision is structural, not a one-off.

## Root cause (verified in code)
- `apps/api/src/databases/databases.service.ts` → `uniqueSlug(workspaceId, name)`:
  `slugify(name)` + `_${i+1}` on collision. Positional, meaningless, order-dependent.
- Uniqueness is **per workspace**: `databases_workspace_slug_uq` on `(workspaceId, api_slug)`
  (`schema.ts:160`). Same db name in different spaces ⇒ collision ⇒ `_2`.
- **Spaces have no `slug`** (only `name`); only `workspaces.slug` exists (`schema.ts:72`).
- `packages/mcp/src/resolve.ts` → `resolveDatabase`: matches `id || name || apiSlug`; a bare
  **name** hit returns the **first** match — no ambiguity guard on exact-name. MCP
  `list_databases` doesn't even show which **space** a db is in, so `projects` vs `projects_2`
  are indistinguishable to an agent.
- REST addresses databases by **UUID** (`:db` = id, `assertAccess` uses `databases.id`), so the
  REST path is safe; the risk is entirely in **name/slug resolution**.
- Same class of bug for **fields**: `fields_database_api_name_uq` on `(database_id, api_name)`
  also `_2`-suffixes (`status`, `status_2`) — analogous, lower urgency (see Scope).

## Proposed design (decisions to lock)
Goal: every database has a **stable, meaningful, collision-free** qualified reference that a
human/agent can read and that never silently points at the wrong thing.

1. **Give spaces a slug.** Add `spaces.slug` (unique per workspace), auto-derived from name,
   editable, immutable-by-default once set. Backfill existing spaces.

2. **Qualify the database reference by space.** Canonical reference =
   `"{space_slug}/{database_slug}"` (or `.` separator) — e.g. `client-work/projects` and
   `storyfunnels/projects`. Two same-named DBs in different spaces both keep the clean slug
   `projects`; the space qualifier disambiguates. Change db-slug uniqueness scope from
   `(workspace, slug)` → **`(space, slug)`**, dropping the `_2` where it's now unambiguous.
   (Decide: keep a workspace-wide short prefix too, e.g. `jcm/client-work/projects`? The user
   floated a 4-letter workspace slug — evaluate whether we need cross-workspace-unique refs or
   whether workspace is always implied by the token/scope. Leaning: workspace is implied by the
   PAT/session, so `space/db` is enough.)

3. **Safe resolution (the actual footgun fix).** In the MCP **and** any name/slug API lookup:
   - Accept the qualified form `space/db` and resolve exactly.
   - A **bare** name/slug that matches >1 database must **error** and list the candidates with
     their space + qualified slug — never silently pick the first.
   - `list_databases` / `describe_database` / get_started must surface each db's **space** and
     **qualified slug** so agents use the unambiguous form.

4. **Stability rules.** Slugs are set at create and **don't change on rename** (already true
   for db slug). Moving a db between spaces changes its qualified ref — surface that as a
   deliberate, warned action (it's effectively a rename of the reference).

## Migration
- Add `spaces.slug`, backfill (slugify name, dedupe per workspace).
- Recompute db slugs under the new `(space, slug)` scope; drop now-unneeded `_N` suffixes.
- This **changes existing slugs** (e.g. `projects_2` → `projects` under a different space).
  Acceptable now (pre-integration), but do it as one deliberate migration and log the old→new
  map. After this lands, treat slugs as a stable public contract.
- Separately: flag/merge the actual **duplicate** "Projects" in JCM (data cleanup, not schema).

## Acceptance
- [ ] Spaces have slugs; db slugs are unique **per space** and clean (no `_2` from cross-space
      name reuse).
- [ ] Canonical reference `space/database` works across REST-by-slug (if exposed), MCP, and UI.
- [ ] A bare ambiguous name/slug **fails loudly** with the qualified candidates — never resolves
      to an arbitrary first match.
- [ ] MCP `list_databases` / `describe_database` / `get_started` show space + qualified slug.
- [ ] Migration recomputes existing slugs deterministically and records the old→new mapping.

## Scope / follow-ups
- **Fields**: the same `_2` collision exists for `api_name` within a database. Not namespaced
  (fields are already db-scoped), but the positional suffix is fragile — separate ticket to make
  field api_names deterministic + ambiguity-safe in resolution.
- **create_relation / schema MCP tools (MN-146)** must emit and accept the qualified form.
